## 2. 一期业务范围

```text
创建知识库 → 上传文档 → 自动切片并入库 → 按问题检索并回答
```

| 步骤 | 用户侧操作 | 系统行为（一期） |
|------|------------|------------------|
| 1 | 调用接口创建知识库 | 保存名称与切片/检索等配置（`config_json` 快照） |
| 2 | 上传 txt、md、docx、xlsx、xls | **同步处理**（HTTP 阻塞至 `SUCCESS` / `FAILED`） |
| 3 | — | 解析正文 → 规则切片 → 按策略 AI 重切 → 向量化写入 pgvector |
| 4 | 查看切片列表（可选） | 检查分段正文是否可用 |
| 5 | 提问 | 向量检索 TopK → 大模型生成回答（或仅返回检索摘要） |

**支持格式**：`txt` / `md` / `docx` / `xlsx` / `xls`（**PDF 规划在二期**）

**一期没做**：原始文件对象存储、异步任务与进度轮询、知识库/文档删除 API、PDF、Rerank/混合检索配置 API。

### 5.2 创建知识库请求体说明（`POST /knowledge-bases`）

创建后，除 `name` / `description` 外，其余字段与 `application.yaml` 默认值合并，写入 `knowledge_base.config_json` **配置快照**；之后**上传文档、问答**均按该快照执行（创建接口本身不处理文件）。

**请求示例**（联调常用：`aiMode=never` 纯规则切片）：

```json
{
  "name": "kb-test-20260520-153630",
  "description": "API test",
  "chunking": { "mode": "hybrid", "aiMode": "never" },
  "rule": {
    "maxChars": 1200,
    "minChars": 80,
    "overlap": 80,
    "plainSeparators": ["\n\n", "\n"],
    "markdownSeparators": ["\n## ", "\n\n", "\n"],
    "sentenceBoundaryFallback": true
  },
  "quality": { "scoreThreshold": 70 },
  "ai": {
    "chunkModel": "qwen-plus",
    "maxCallsPerDoc": 1,
    "maxInputTokens": 8000,
    "retryOnParseError": 1
  },
  "embedding": { "model": "text-embedding-v3" },
  "retrieval": { "topK": 3, "scoreThreshold": 0.3 }
}
```

独立文档：**[创建知识库接口参数.md](创建知识库接口参数.md)**（完整参数表、默认值、校验与场景）。  
样例 JSON：[api-samples/create-knowledge-base-request-full.json](api-samples/create-knowledge-base-request-full.json)

#### 顶层

| 字段 | 示例 | 含义 |
|------|------|------|
| `name` | `kb-test-20260520-153630` | 知识库名称（必填），仅标识与展示 |
| `description` | `API test` | 可选描述，写入 `knowledge_base.description` |

#### `chunking` — 切片策略

| 字段 | 示例 | 含义 |
|------|------|------|
| `mode` | `hybrid` | 一期固定 **混合模式**：规则切片为主，按需千问重切 |
| `aiMode` | `never` | 是否调用千问语义重切，见下表 |

| `aiMode` | 上传时行为 |
|----------|------------|
| `never` | 仅规则切片（**T1**）；`smartChunk=true` **无效** |
| `auto` | 规则后评估：质量分低于 `quality.scoreThreshold` → **T2**；整篇仅 1 段 → **T4** |
| `always` | 每篇文档千问重切（**T0**） |

#### `rule` — 规则切片（`RuleChunker`）

| 字段 | 示例 | 含义 |
|------|------|------|
| `maxChars` | `1200` | 单块字符上限；仍超长则滑动窗口继续切 |
| `minChars` | `80` | 单块下限；过短片段与相邻段合并 |
| `overlap` | `80` | 相邻块重叠字符数，后块头部拼接前块尾部（利于检索上下文） |
| `plainSeparators` | `["\n\n","\n"]` | txt / docx / xlsx 等**非 md** 按序尝试的分段符 |
| `markdownSeparators` | `["\n## ","\n\n","\n"]` | **仅 .md** 优先按二级标题、空行、换行切 |
| `sentenceBoundaryFallback` | `true` | 按 `maxChars` 硬切时向前找句号/问号等，减少断句 |

文档 profile 由系统自动识别：`.md` → markdown 分隔符，其余 → plain。

#### `quality` — 规则切片质量评估

| 字段 | 示例 | 含义 |
|------|------|------|
| `scoreThreshold` | `70` | 质量分 0–100；**仅 `aiMode=auto` 时**，低于该分触发 T2。`never` 时不上传阶段不生效，但会保存在快照中 |

#### `ai` — 千问语义重切（`SemanticResplitService`）

| 字段 | 示例 | 含义 |
|------|------|------|
| `chunkModel` | `qwen-plus` | 语义重切对话模型（实际提供方由 DashScope / Ollama profile 决定） |
| `maxCallsPerDoc` | `1` | 单文档最多调用千问次数 |
| `maxInputTokens` | `8000` | 送入模型原文 token 上限，超出截断 |
| `retryOnParseError` | `1` | 切片 JSON 解析失败时的重试次数 |

`aiMode=never` 时上传**不执行**上述 AI 配置，但会写入快照；改为 `auto`/`always` 后生效。

#### `embedding` — 向量化

| 字段 | 示例 | 含义 |
|------|------|------|
| `model` | `text-embedding-v3` | 切片入库与**问题向量化**使用的 Embedding 模型；须与库表向量维度（默认 1024）一致 |

#### `retrieval` — 问答检索（`RagChatService`）

| 字段 | 示例 | 含义 |
|------|------|------|
| `topK` | `3` | 每次问答最多取相似切片条数 |
| `scoreThreshold` | `0.3` | 余弦相似度下限（0–1），低于阈值的切片不进入 prompt |

仅影响 `POST .../chat`，不影响上传切片。

#### 示例配置的整体效果（一句话）

创建 API 测试库 → 上传时**只做规则切片**（约 1200 字/块、重叠 80、按段落/标题切）→ `text-embedding-v3` 入库 → 问答最多 3 段、相似度 ≥ 0.3。

#### 与上传接口的关系

| 上传参数 | 在 `aiMode=never` 库下 |
|----------|-------------------------|
| `file` | 文档内容（一期不落盘，仅内存解析） |
| `smartChunk` | 即使为 `true` 也**不会**调千问 |

未传的子字段由 `KnowledgeBaseConfigMerger` 与 `application.yaml` 默认值合并。

## 6. 切片与 AI 策略（速查）

创建知识库时写入的 `chunking` 配置在上传时生效，流水线大致为：

```text
解析(R0) → 规则切片(R2) → 质量评估 → [按策略 AI 重切] → 校验 → 向量化 → 入库
```

| 配置 / 上传参数 | 行为 |
|-----------------|------|
| `aiMode=never` | 仅规则切片（T1） |
| `aiMode=always` | 全量 AI 切片（T0） |
| `aiMode=auto` | 质量分/弱边界触发 T2 或 T4 |
| `smartChunk=true` | `aiMode≠never` 时强制 T8；`never` 时无效 |

规则切片要点：`RuleChunker` 按 profile 分隔符切分 → 句界回退 → overlap；详见代码注释与设计稿 [archive/chunking.md](archive/chunking.md)、[archive/hybrid-chunking.md](archive/hybrid-chunking.md)。

## 7. 数据保存说明

| 内容 | 一期是否保存 | 存储位置 |
|------|--------------|----------|
| 用户上传的**原始文件** | ❌ **不保存** | 仅内存 `MultipartFile` 解析；`storage_key` 恒为 NULL |
| 文档元数据 | ✅ | `document`（文件名、状态、切片数、`ai_triggered` 等） |
| 切片正文 | ✅ | `chunk.text_content` |
| 向量 | ✅ | `chunk.embedding`（列表接口不返回向量） |
| 知识库配置 | ✅ | `knowledge_base.config_json` |

**结论**：一期可检索的是**切片后的文本**，无法通过系统下载原始 docx/xlsx（二期规划对象存储）。

## 8. 主要风险与质量问题

与「接口是否调通」无关，当前最影响可用性的问题：

| # | 问题域 | 影响 |
|---|--------|------|
| 1 | **文件解析**正文与原件不一致（尤其 xlsx/docx） | 脏数据进入切片与向量库 |
| 2 | **混合切片**段边界不合理（过长/过短/断句/重复） | 检索引用质量差 |
| 3 | **AI 重切**常 `aiFallback=true`，回退规则结果 | 智能切片效果难预期 |
| 4 | 团队**首次落地 LLM 切片**，调试周期长 | 排期需预留 |

问题现象、涉及类路径、建议攻关项 → 见 **[详细版 §9](一期开发进度-详细版.md#9-目前存在的问题)**。
