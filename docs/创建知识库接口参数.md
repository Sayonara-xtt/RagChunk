# 创建知识库接口参数说明

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/v1/knowledge-bases` |
| Content-Type | `application/json` |
| 响应码 | `201 Created` |
| 文档版本 | 1.0 |
| 更新日期 | 2026-05-22 |

---

## 1. 接口说明

创建知识库时，服务端将请求体与 **`application.yaml` 中 `ragchunk.*` 默认值**合并（`KnowledgeBaseConfigMerger`），生成完整配置快照写入 `knowledge_base.config_json`。

- **创建接口不处理文件**；文档上传走 `POST /api/v1/knowledge-bases/{kbId}/documents`。
- 之后该库下所有**上传、向量化、问答**均按此快照执行，修改需新建库（**更新库 API 未实现**）。
- 调试入口：Scalar `http://localhost:8080/scalar`（或 `/doc.html` 重定向）→ **知识库 → 创建知识库**。

```json
  },

    "maxInputTokens": 8000,

  },
  "embedding": {
    "model": "text-embedding-v3"
  },
  "retrieval": {
    "topK": 3,
    "scoreThreshold": 0.3
  },
  "qa": {
    "scheme": 1,
    "rewriteMinScore": 0.35,
    "maxRewriteQueries": 2,
```

## 3. 响应示例

```json
{
  "id": "kb_93154bcde024",
  "name": "kb-test-20260520-153630",
  "description": "API test",
  "status": "READY",
  "config": { },
  "createdAt": "2026-05-20T07:36:31.068083Z"
```

## 4. 参数一览

### 4.1 顶层

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `name` | string | **是** | — | 知识库名称，最长 **64** 字符，首尾空格会 trim |
| `description` | string | 否 | `null` | 描述，写入 `knowledge_base.description` |

### 4.2 `chunking` — 切片策略

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `mode` | string | 否 | `hybrid` | 当前仅支持 **hybrid**（规则为主 + 按需千问重切） |
| `aiMode` | string | 否 | `auto` | 千问介入策略，见 §5 |

### 4.3 `rule` — 规则切片（`RuleChunker`）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `maxChars` | int | 否 | `1200` | 单块最大字符数；超长则继续切分 |
| `minChars` | int | 否 | `80` | 单块最小字符数；过短片段与相邻段合并 |
| `overlap` | int | 否 | `80` | 相邻块重叠字符数 |
| `plainSeparators` | string[] | 否 | `["\n\n","\n"]` | txt / docx / xlsx 等**非 md** 文档的分段符（按顺序尝试） |
| `markdownSeparators` | string[] | 否 | `["\n## ","\n\n","\n"]` | **仅 .md** 使用的分段符 |
| `sentenceBoundaryFallback` | boolean | 否 | `true` | 按 `maxChars` 硬切时，是否向前查找句号/问号等再切断 |

**校验**：`maxChars` 必须 ≥ `minChars`，否则 `400`。

**Profile**：扩展名 `.md` 用 `markdownSeparators`，其余用 `plainSeparators`（`ChunkProfileDetector` 自动识别）。

### 4.4 `quality` — 规则切片质量评估

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `scoreThreshold` | int | 否 | `70` | 质量分 0–100；**仅 `aiMode=auto`** 时，低于该分触发千问重切（T2） |

### 4.5 `ai` — 千问语义重切（`SemanticResplitService`）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `chunkModel` | string | 否 | `qwen-plus` | 语义重切使用的对话模型 |
| `maxCallsPerDoc` | int | 否 | `1` | 单个文档允许的最大千问调用次数 |
| `maxInputTokens` | int | 否 | `8000` | 送入模型的原文 token 上限，超出截断 |
| `retryOnParseError` | int | 否 | `1` | 模型返回切片 JSON 解析失败时的重试次数 |

`aiMode=never` 时上传阶段**不调用**千问，但配置仍会持久化。

### 4.6 `embedding` — 向量化

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `model` | string | 否 | `text-embedding-v3` | 切片入库与**用户问题**向量化使用的模型 |

向量维度由服务端 `ragchunk.embedding.dimensions` 控制（默认 **1024**），须与库表 `chunk.embedding` 一致。

### 4.7 `retrieval` — 问答检索（`RagChatService`）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `topK` | int | 否 | `3` | 每次问答返回的最大切片条数，范围 **1–20** |
| `scoreThreshold` | double | 否 | `0.5` | 余弦相似度下限（0–1），低于阈值的切片不进入上下文 |

**说明**：请求示例中 `scoreThreshold: 0.3` 会覆盖默认 `0.5`。仅影响 `POST .../chat`，不影响上传。

### 4.8 `qa` — 智能问答编排（`ChatOrchestrator`）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `scheme` | int | 否 | `1` | 见 [智能问答方案.md](智能问答方案.md) 编码对照：`1` 纯应用流水线 · `2` 协作渐进检索 · `3` 协作全量改写 · `5` Agent 工具检索 |
| `rewriteMinScore` | double | 否 | `0.35` | **协作渐进检索**：最高分低于此值触发改写（0–1） |
| `maxRewriteQueries` | int | 否 | `2` | 改写产出检索句条数（1–5） |
| `maxLlmCalls` | int | 否 | `2` | 单次问答 LLM 次数上限（1–5） |
| `maxSearchRounds` | int | 否 | `2` | 单次问答检索轮次上限（1–5） |
| `agentMaxIterations` | int | 否 | `3` | **Agent 工具检索**：对话轮次（1–10） |
| `agentMaxToolCallsPerRound` | int | 否 | `1` | **Agent**：每轮 `search_kb` 次数（1–3） |
| `agentAllowRelaxThreshold` | boolean | 否 | `true` | **Agent**：tool 是否允许放宽相似度阈值 |

详细流程见 **[智能问答方案.md](智能问答方案.md)**。

## 5. `chunking.aiMode` 与上传行为

上传文档：`POST /api/v1/knowledge-bases/{kbId}/documents`（`multipart/form-data`，**HTTP 202**）

| 表单字段 | 说明 |
|----------|------|
| `file` | 单文件（与 `files` 二选一） |
| `files` | 多文件批量 |
| `smartChunk` | 默认 `false`，见下文 |
| `sourceType` | 可选；单文件默认 `API_SINGLE`，多文件默认 `LOCAL_BATCH` |

**受理方式**：请求线程 **流式** 写入原件存储（`ragchunk.oss.local-root`），再异步入库；**不会**在响应中直接返回 `chunkCount`，须轮询 `GET .../documents/{docId}` 至 `status=SUCCESS`。详见 **[异步上传与OSS.md](异步上传与OSS.md)**。

| `aiMode` | 上传时切片行为 | `document.ai_trigger_id` |
|----------|----------------|---------------------------|
| `never` | 仅规则切片 | `T1` |
| `always` | 每篇文档千问重切 | `T0` |
| `auto` | 规则切片 + 质量评估，按需重切 | `T2` / `T4` 或空 |

**auto 触发条件**（`AiChunkTrigger`）：

1. 规则切片质量分 **<** `quality.scoreThreshold` → **T2**
2. 整篇文档规则切片后 **仅 1 段**（长文单段）→ **T4**
3. 否则不调千问

**上传参数 `smartChunk`**（表单，默认 `false`）：

| 条件 | 效果 |
|------|------|
| `smartChunk=true` 且 `aiMode` 为 `auto` / `always` | 强制千问重切 → **T8** |
| `aiMode=never` | `smartChunk` **无效**，始终仅规则切片 |

## 6. 配置生效阶段

| 配置块 | 创建库 | 上传文档 | 问答 |
|--------|--------|----------|------|
| `name` / `description` | ✅ | — | — |
| `chunking` / `rule` / `quality` / `ai` | 写入快照 | ✅ 切片流水线 | — |
| `embedding` | 写入快照 | ✅ 切片向量化 | ✅ 问题向量化 |
| `retrieval` | 写入快照 | — | ✅ TopK + 相似度过滤 |
| `qa` | 写入快照 | — | ✅ 问答编排方案与次数上限 |

流水线（上传）：

```text
[HTTP] 流式归档原件 → 202 返回 docId
[后台] 解析 → 规则切片 → 质量评估 → [AI 重切] → 向量化 → SUCCESS
```

## 7. 常见配置组合

| 场景 | 建议 `aiMode` | 说明 |
|------|---------------|------|
| 联调 / 可复现测试 | `never` | 不耗 LLM，仅验证解析与规则切片 |
| 生产试运行（省成本） | `auto` + `scoreThreshold: 70` | 质量差或长文单段才调千问 |
| 问答平衡召回 | `qa.scheme: 2` | 协作渐进检索，难问才多 1 次 LLM |
| 问答纯应用 | `qa.scheme: 1` | 纯应用流水线，最低延迟 |
| 强依赖 AI 切片 | `always` | 每篇都调千问，成本高、需配置 API Key |
| 单次强制 AI | 库级 `auto` + 上传 `smartChunk=true` | 对应 T8 |

## 8. 错误码

| HTTP | 原因示例 |
|------|----------|
| `400` | `name` 为空或超过 64 字；`aiMode` 非法；`qa.scheme` 非 1/2/3/5；`maxChars < minChars`；`topK` 不在 1–20 |
| `201` | 创建成功 |
